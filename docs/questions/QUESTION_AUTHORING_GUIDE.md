# PyPractice 题目编写规范

> 本文是出题 Agent、审题 Agent 和项目实现者的共同约束。当前只完成题库规划与首批 30 题蓝图，尚未录入可发布题目。首批 30 题在阶段 2 实现，完整 200 题在后续阶段完成。

## 1. 目标与原则

- 面向正在学习 Python 基础语法的学习者，强调短小、可重复、反馈明确的练习，不演变为算法竞赛。
- 单题集中考查 1-2 个知识点；不得偷用明显超出当前章节的语法。
- 题面、示例、参考答案、测试、提示和解析必须一致，参考答案必须真实执行并通过全部测试。
- 不依赖网络、宿主文件、外部进程或未承诺的第三方包；文件题只使用执行沙箱中的虚拟文件。
- 隐藏测试用于验证公开要求的边界，不得加入题面未说明的新规则。
- 出题与审题职责分离；未通过独立审核的题目保持 `draft` 或 `review`。

## 3. 题型与难度矩阵

题型：`coding` 编程、`fillBlank` 代码补全、`debug` 纠错、`output` 输出预测、`multipleChoice` 选择、`codeOrder` 代码排序、`miniProject` 小型综合题。

| 题型 | Intro | Basic | Skilled | Combined | 合计 |
|---|---:|---:|---:|---:|---:|
| coding | 10 | 30 | 20 | 10 | 70 |
| fillBlank | 10 | 12 | 7 | 1 | 30 |
| debug | 6 | 10 | 7 | 2 | 25 |
| output | 10 | 10 | 5 | 0 | 25 |
| multipleChoice | 10 | 7 | 3 | 0 | 20 |
| codeOrder | 2 | 5 | 6 | 2 | 15 |
| miniProject | 0 | 1 | 6 | 8 | 15 |
| **合计** | **48** | **75** | **54** | **23** | **200** |

难度口径：

- `intro`：单一知识点、提示充分、代码短、边界简单。
- `basic`：独立写出基础语法，组合相邻知识点，包含常见边界。
- `skilled`：需要选择合适语法或数据结构，提示较少、测试更丰富。
- `combined`：综合多个基础知识点，强调程序结构，但不要求竞赛算法。

难度是学习负荷，不是“坑题”程度。隐藏测试不得把题目实际难度抬高。

## 4. 标准字段

```ts
{
  id: 'ch01-q001',
  slug: 'variable-assignment',
  title: '给变量赋值',
  chapterId: 'chapter-01',
  knowledgePointIds: ['variable-assignment'],
  type: 'fillBlank',
  difficulty: 'intro',
  description: '说明输入、输出和目标，不泄露解法。',
  requirements: ['每项都是可验证的明确要求。'],
  starterCode: '',
  solutionCode: '',
  promptCode: undefined,
  examples: [
    { id: 'ch01-q001-e01', input: '', expectedOutput: '', visible: true, compareMode: 'lineTrimmed' }
  ],
  hiddenTests: [
    { id: 'ch01-q001-h01', input: '', expectedOutput: '', visible: false, compareMode: 'lineTrimmed' }
  ],
  testHarnessCode: undefined,
  choices: undefined,
  correctChoiceIds: undefined,
  codeBlocks: undefined,
  correctCodeOrder: undefined,
  hints: ['提示知识点', '提示步骤', '指出关键位置但不直接给答案'],
  explanation: '解释关键语法、思路和常见错误。',
  estimatedMinutes: 5,
  status: 'draft',
  createdAt: 'YYYY-MM-DD',
  version: 1
}
```

字段要求：

- `id` 全局唯一且稳定，`chapterId` 和 `knowledgePointIds` 必须引用有效目录项。
- `coding`、`fillBlank`、`debug`、`miniProject` 必须有可运行的 `solutionCode`。
- `output` 的待分析代码放在 `promptCode`，正确答案保存为结构化答案或专用字段，不与 `starterCode` 混用。
- `multipleChoice` 至少 3 个语义平行且不含歧义的选项；`correctChoiceIds` 不得为空。
- `codeOrder` 的块 ID 唯一；存在多个等价顺序时必须列出全部等价答案或改写题目。
- 非可选字段始终存在；空字符串/空数组仅在业务语义确实为空时使用，不能绕过校验。

## 5. 执行与测试协议

### 5.1 判题方式

- 标准输入/输出题直接使用 `input` 与 `expectedOutput`。
- 要求定义函数、返回值或对象状态的题，由系统附加用户不可编辑的 `testHarnessCode` 调用并输出结构化结果。
- 文件题通过 `virtualFiles` 提供路径、UTF-8 内容和运行后断言，每次测试使用全新沙箱。
- 选择、排序、输出预测使用专用结构判定，不伪装成 stdout。
- harness/fixture 协议未实现前，相应题目必须保持草稿，不能用空测试占位。

### 5.2 测试设计

每个可执行题至少有 1 个可见示例和 1 个隐藏测试。建议基础题 2-3 个隐藏测试，熟练/综合题 3-5 个，覆盖：

1. 典型输入；
2. 题面承诺范围内的合法边界，如 0、空值、单元素、重复值；
3. 分支与循环端点；
4. 空格、大小写、换行和精度等格式要求；
5. 至少一个典型错误答案可被拦截。

测试 ID 唯一；`visible` 与所属集合一致；预期结果应由参考实现生成或复核，不允许凭直觉填写。

### 5.3 比较模式

- `exact`：仅将 CRLF/CR 统一为 LF；空格、大小写和终止换行必须完全一致。stdout 的 `expectedOutput` 必须显式包含终止 `\n`。
- `trimmed`：仅忽略整体首尾空白。
- `lineTrimmed`：统一换行并忽略每行末尾空格，适合一般文本输出。
- `numeric`：只用于数值误差，必须指定 `numericTolerance`。
- `unorderedLines`：仅当题面明确不要求顺序时使用。
- `custom`：记录明确、可测试的判定规则，并单独测试比较器。

若格式本身是考点，不得使用过度宽松的比较模式。标准输入不应包含“请输入”等提示词，除非题面明确要求输出提示。

### 5.4 安全边界

- 单次参考答案应远低于默认 3 秒限制，默认最大输出 20,000 字符。
- 禁止网络、`socket`、`subprocess`、多进程、宿主绝对路径、动态安装包和无限等待输入。
- 虚拟文件路径必须是白名单内相对路径，测试之间互不污染。
- 用户代码、harness、测试和题面数据分别传递；不得把隐藏测试当成可公开展示内容。

## 6. 提示、解析与文字规范

- 通常提供 3 级提示：知识方向 -> 解题步骤 -> 关键位置；不直接粘贴完整答案。
- 解析说明思路、关键语法、边界与常见错误，不能只重复代码。
- 使用清晰、简短、无歧义的中文；输入输出格式、索引起点、排序规则、精度和异常策略必须写明。
- 不使用依赖性别、地域或文化背景的无关叙事；示例数据不包含真实个人信息。

## 7. 审核流程与发布门槛

1. 出题 Agent 按章节配额生成 `draft`，自检字段、答案和测试。
2. 自动校验 Agent 检查 Schema、ID、配额、引用和字段互斥关系。
3. 执行标准答案与典型错误答案，核对所有比较模式、harness 和 fixture。
4. 独立审题 Agent 检查知识点、难度、歧义、边界、提示与解析，不直接沿用出题结论。
5. 修订并复审；只有全部通过后将状态改为 `approved`。

发布前必须同时通过：Schema 校验、题库统计、参考答案执行、典型错误拦截、人工审核、单元测试、lint 和构建。
